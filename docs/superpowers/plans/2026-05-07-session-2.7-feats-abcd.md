## Phase 0 — Pre-flight decisions (USER INPUT REQUIRED, NO CODE)

The cross-cutting review surfaced three decisions that must land before code begins. Phase 1 cannot start until the user has answered all three.

### Q0.1 — Add a new `WorkoutStatus.abandoned` value, or reuse `skipped`?

Source: Feat C open question 2; review §6 (top blocker).

**Why it matters:** Feat C shifts the start date. Past planned workouts that the athlete never did need a status. `skipped` (current enum) implies user choice; `abandoned` is more honest but adds a schema migration that ripples through Feat A's state guards (must reject log-completed on `abandoned` rows, like `done`/`skipped` do today), Feat B's adherence math (`abandoned` should NOT count as a "skip" against the on-plan %), and Feat D's status-pill rendering (red? grey? cutback-blue?).

**Default if no answer:** Add `abandoned`. Cleaner semantics, the migration is cheap.

### Q0.2 — `MIN_WEEKS_TO_RACE` thresholds

Source: Feat C open question 5; review §6.

Feat C refuses a start-date shift if it would compress build-up below a per-race minimum. The spec proposes `{MCM: 16, Disney: 8, Delaware: 9}`. The user must confirm or override. These are guesses; the user is the athlete.

**Default if no answer:** Use the spec's proposed values; revisit after dogfooding.

### Q0.3 — Font token names: `mono` / `monoBold` (typeface) vs `body` / `bodyBold` (role)?

Source: Feat D open question 2; review §6.

Renaming after ~30 file edits is painful. Feat D used `mono` / `monoBold`. If the user wants role-named, change now.

**Default if no answer:** `mono` / `monoBold` (matches Feat D as written).

### Phase 0 deliverable

A short note in `docs/superpowers/plans/2026-05-07-session-2.7-decisions.md` capturing the three answers + any minor inconsistency fixes (Feat C §5 vs §6 — drop the `/preview` sibling in favor of `?dry_run=true`). Commit this BEFORE Phase 1.
