## Q0.1 — Past planned-but-undone workouts on start-date shift

**Decision:** Reseed semantics. **Do NOT add `WorkoutStatus.abandoned`.** Keep enum at `planned | moved | skipped | done`.

User's words:
> "When I shift the start date, I'm recasting the program; completed workouts remain, but previously planned but incompleted are just recast into the new plan or removed from the equation. Keep completed workouts so long as they fall within the range of the new start date. For example, I started literally this week (Tuesday). The original program had me start 3 weeks ago."

### What this changes

The original Feat C spec recommended **Option α: date delta migration** with `abandoned` as an open question. The user has chosen **Option β: reseed cycles + plan from a new anchor** — the option Feat C explicitly parked as v1.1.

Concrete behavior of `POST /plan/start-date` after this decision:

1. Validate `new_start_date >= today`. (See Q0.2 — no minimum-weeks-to-race refusal.)
2. **Keep** all `completed_workouts` rows whose `activity_date >= new_start_date` AND `<= last_race_date`. Drop the rest from the active program (they remain in the DB as historical data; reconciliations point at them but are no longer surfaced via plan endpoints).
3. **Keep** `planned_workouts` rows with `status IN ('done', 'skipped')` whose `scheduled_date >= new_start_date` — these are the user's record of what they actually did / explicitly chose to skip.
4. **Delete** `planned_workouts` rows with `status IN ('planned', 'moved')` belonging to cycles being reseeded. This includes rows with `original_snapshot_json` user edits — those edits are lost on reseed (acceptable per user direction).
5. **Re-emit** fresh `planned_workouts` rows for cycle 1 (and any later cycle whose start_date the reseed pushed past) using the parser parametrized with `new_start_date`.
6. **Discard** all pending `agent_messages` proposals (`state='pending'`) — they reference deleted rows. Apply `state='discarded'`, `discard_reason='plan_shifted'`.
7. Cycles 2 and 3 are NOT reseeded — they anchor to fixed race dates (Disney 2027-01-10, Delaware 2027-04-11). Only the boundary date between cycles may shift if cycle 1 now ends earlier.
8. Write a `plan_history` row capturing `{action: "start_date_reseed", old_start, new_start, completed_kept, planned_dropped, ...}`.
9. Bust caches via `invalidate_for_athlete`.

### Parser parametrization

`app/seed/plan_parser.py:11-36` currently hardcodes cycle start dates. To support reseed, the parser must accept a `start_date` override for cycle 1 and compute week schedules from there. This is a contained change — single function param + propagating through the date arithmetic. New sub-task in the plan.

### Variable cycle 1 length

If the user reseeds today (2026-05-06) and Cycle 1 race is 2026-10-25, that's ~25 weeks of build-up. The PLAN.md template structures Cycle 1 over 28 weeks. The parser must handle a shorter cycle by dropping the earliest weeks (1, 2, 3 in this example) — i.e., the athlete starts at "template week 4" of the prescribed sequence. Documented behavior in plan task 2.C.

## Q0.2 — `MIN_WEEKS_TO_RACE` thresholds

**Decision:** No minimum. The user accepts whatever shift they ask for, even one that compresses training to 1 week before the race.

User's words:
> "There is no minimum, that is my choice, just program the training program."

### What this changes

Feat C §5 had a 400-error refusal when a shift would compress build-up below `MIN_WEEKS_TO_RACE = {1: 16, 2: 8, 3: 9}`. **Drop that validation.** The endpoint accepts any `new_start_date >= today` and emits whatever fits.

Soft warning is still useful — the dry_run impact preview should call out unusually short windows (e.g., "WARNING: 4 weeks until MCM"). UI text only; no 400 from the backend.
