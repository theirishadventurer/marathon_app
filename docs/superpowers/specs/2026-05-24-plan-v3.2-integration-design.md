# Plan v3.2 Integration — Design

**Date:** 2026-05-24
**Goal:** Integrate the new `marathon_trilogy_plan_v3.2.md` (Phase 1 only, MCM build) into the app's seed pipeline, replacing the current v2.0 Phase 1 content, with Phase 2 (Disney) and Phase 3 (Delaware) preserved from v2.0 but tagged as preliminary until they're re-anchored post-MCM.
**Driver:** Plan kickoff is Monday 2026-05-25 (tomorrow). Current seed has cycle 1 starting `2026-04-13` with v2.0's 6-run/week structure; new plan is 4-5 runs/week with strength-only Mon/Fri.

## Decisions made during brainstorming

| Decision | Choice | Why |
|---|---|---|
| Slider/flexibility modeling | **Deterministic days, document flexibility in body text** | Drag-to-move + edit-workout UX already handles real-life shuffles. Avoids schema and mobile work for tonight. |
| Cycles 2/3 policy | **Keep v2.0 content, tag every workout description with "PRELIMINARY — "** | Program tab stays visually populated. Tag forces awareness that data is provisional. |
| Workout type granularity | **Reuse existing `WorkoutType` enum values, encode flavor in body text** | Avoids Alembic migration on a native Postgres enum. `WorkoutCard` already renders body text. |

## §1 — Day-of-week mapping (deterministic policy)

| Day | WorkoutType | Notes |
|---|---|---|
| Mon | `strength_a` | Body text: "intro" variant for weeks 1-2, full Strength A for weeks 3+ |
| Tue | `easy` | Trail run. Body text mentions "Trail (Perkiomen)" + distance |
| Wed | `tempo` or `mp_long` | **The quality day.** `tempo` for tempo/threshold prescriptions, `mp_long` for MP-effort prescriptions |
| Thu | `easy` | Trail run, same as Tue |
| Fri | `strength_b` | Body text: "intro" variant for weeks 1-2, full Strength B for weeks 3+ |
| Sat | `long` | Slider default = Sat. Includes fueling note per progression table |
| Sun | `recovery` | 2-3mi very easy. **Changed from current `rest` on most weeks.** |

**Flexibility lives outside the schema:** athlete uses drag-to-move (Week tab) or edit-workout to shuffle the floating quality day or slider weekend.

## §4 — Re-seed strategy

The rewrite changes week structure (different workout types and counts per week), so `seed_plan`'s idempotent upsert *probably* leaves stale rows. Cleanest: wipe and re-seed.

**Local (tonight):**
```powershell
docker compose down -v
docker compose up -d --build
docker compose exec -T api alembic upgrade head
docker compose exec -T api python -m app.seed.load_plan
```

**Railway (when deploying):**
- Fresh deploy → first boot runs `alembic upgrade head` per `railway.json` `startCommand` → run `python -m app.seed.load_plan` in shell per deploy runbook §2.5. Empty DB, so no stale rows.
- For a future re-seed against a populated Railway DB: write a small `app/seed/reset_planned.py` that deletes only `planned_workouts WHERE status = 'planned'` for a given athlete, preserving `completed_workouts`, `reconciliations`, and `plan_history`. **Out of scope tonight** (fresh deploy doesn't need it).

## §5 — Validation

1. **Backend tests:** `docker compose exec -T api pytest` — all green (104/104 per latest baseline).
2. **Spot-check seed output:**
   ```sql
   SELECT week_number, scheduled_date, type, description_md
   FROM planned_workouts
   WHERE cycle_id = (SELECT id FROM cycles WHERE sequence = 1)
   ORDER BY scheduled_date
   LIMIT 30;
   ```
   Expect 22 weeks × 7 rows = 154 rows for cycle 1. Mon = strength_a, Sat = long, Sun = recovery.
3. **Mobile typecheck:** `cd mobile && npx tsc --noEmit` (no app changes expected; sanity check).
4. **Local smoke:** open app, login, verify:

   - Tap to Mon May 25 — should show "Strength A — intro" with goblet squat / RDL / push-ups body
   - Week view shows new Mon/Fri = strength, Sat = long, Sun = recovery
   - Program tab still shows 3 lanes; P2/P3 weeks show "PRELIMINARY" prefix in card body

## §7 — Risks

| Risk | Mitigation |
|---|---|
| Hand-translation introduces transcription errors (wrong distance, wrong day) | Spot-check via SQL + mobile smoke in §5. Cross-reference against the v3.2 long-run progression table at the end of the source plan. |
| Idempotent seed leaves stale rows from old cycle 1 weeks (e.g., old week 23-28) | Use destructive re-seed in §4 — `docker compose down -v` wipes pgdata, fresh seed has no ghosts. |
| Sun=recovery breaks family expectations (was rest before) | Acceptable — it's an actual change in plan philosophy from v2.0 to v3.2. Recovery runs are explicitly droppable per v3.2 priority order. |
| Phase 2/3 prefix accidentally drops or duplicates | Validate by SQL: every cycle 2 + 3 workout body should start with "PRELIMINARY — ". |
